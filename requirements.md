# 幼儿园教案生成 H5 项目需求文档（草案）

## 1. 项目目标

面向幼儿园老师提供一个轻量 H5 系统，支持在线输入教学信息，并基于大模型 API 智能生成教案。系统以“模板 + 输入内容 + 会话沉淀”为核心，尽量降低老师编写教案的时间成本。

## 2. 产品定位

- 使用对象：幼儿园老师
- 使用终端：移动端优先的 H5 页面，兼容 PC 浏览器
- 核心价值：快速生成结构化教案，沉淀个人模板与历史会话，支持持续迭代与复用
- 页面要求：核心页面需要兼容 PC 端使用，保证会话、生成、查看、编辑等主流程可正常完成

## 3. 核心业务流程

1. 用户注册或登录系统
2. 默认进入首页，会话列表即首页
3. 点击历史会话进入详情页，查看老师提问、AI 回答和当前教案结果
4. 点击“新建会话”进入新会话页
5. 在新会话页先选择模板和大模型
6. 通过底部聊天输入框输入需求，点击发送后开始对话并创建会话
7. AI 返回内容后保存聊天记录，并更新当前会话对应的教案结果
8. 用户可点击“查看教案”进入教案编辑页
9. 历史会话可继续追问，也可随时查看或编辑教案结果

## 4. 功能需求

### 4.1 用户注册 / 登录

#### 功能范围

- 支持用户名 + 密码登录
- 支持注册新账号
- 支持退出登录
- 登录后仅访问自己的模板与会话数据

#### 基本规则

- 用户账号唯一
- 密码加密存储
- 登录后返回登录态，前端通过 token 维持会话

### 4.2 教案生成

#### 页面流程

- 进入新会话页时，先选择模板
- 选择本次使用的大模型
- 在底部聊天输入框中输入本次需求
- 点击发送后调用后端接口
- 后端拼装模板内容、用户输入内容和会话上下文后请求大模型
- 返回 AI 回答内容，并同步落库保存
- 当本轮产出教案时，自动更新当前会话的教案结果
- 用户可点击“查看教案”进入教案编辑页

#### 输入项（第一版建议）

- 用户以自然语言输入为主
- 推荐在消息中包含：
  - 主题
  - 教具
  - 活动目标
  - 适用班级 / 年龄段
  - 活动时长
  - 重点难点
  - 其他补充要求

#### 输出内容（第一版固定格式）

- 活动名称
- 活动目标
- 活动准备
- 活动过程
- 延伸活动
- 注意事项

#### 业务规则

- 每次生成必须选择模板
- 每次生成必须选择一个可用的大模型
- 每次生成必须归属一个会话
- 支持在同一会话中连续生成多轮内容
- 新建会话页不使用底部 tab 导航
- 新建会话页底部固定聊天输入框，发送后开始对话
- 会话详情页需展示老师发送内容和 AI 回答内容
- 聊天气泡中的长文本必须自动换行，不允许超出气泡范围
- 每个会话对应一个当前教案结果
- 生成结果需自动落库保存，便于后续查看
- 历史会话的教案结果支持用户随时手动编辑后保存

### 4.3 大模型配置管理

#### 功能范围

- 后台支持配置多个大模型
- 支持设置默认大模型
- 生成教案时前端展示可用大模型列表并允许选择

#### 配置定义

- 大模型配置用于保存接口提供商、Base URL、模型名称、鉴权信息等
- 第一版采用 OpenAI Compatible（Responses）接口风格
- 模型配置由后台统一维护，可直接通过数据库手动配置
- 前端不提供模型配置编辑能力，只提供展示和选择能力

#### 配置字段建议

- 配置名称
- 提供商类型（固定为 OpenAI Compatible）
- base_url
- api_key
- model_code
- 是否启用
- 是否默认
- 排序值
- 创建时间
- 更新时间

#### 业务规则

- 后端从数据库读取启用状态的大模型配置作为当前可用配置
- 前端生成页展示所有可用模型，用户必须选择一个模型后才能生成
- 系统需要支持一个默认大模型，进入生成页时默认选中
- 每条生成记录需要保存当次使用的大模型配置快照信息
- 如果后续模型配置被修改，不影响历史生成记录回溯
- 前端不提供模型配置新增、编辑、删除能力

### 4.4 模板管理

#### 功能范围

- 新建模板
- 编辑模板
- 删除模板
- 查看模板列表
- 查看模板详情

#### 模板定义

- 模板本质是一段文本提示词，用于约束大模型输出风格、结构和内容要求
- 模板仅当前用户可见
- 模板可设置名称，便于选择

#### 模板字段建议

- 模板名称
- 模板内容
- 创建时间
- 更新时间
- 所属用户 ID

### 4.5 会话管理

#### 功能范围

- 新建会话
- 查看会话列表
- 查看会话详情
- 在已有会话中继续生成
- 删除会话

#### 会话定义

- 会话用于组织一次或多次连续生成记录
- 会话下保存老师消息、AI 回复、所选模板、所选模型、当前教案结果、时间信息
- 首页直接展示会话列表
- 会话支持从列表进入并继续生成新内容
- 会话详情以聊天记录形式展示历史上下文
- 历史会话可随时查看或编辑当前教案结果

#### 会话字段建议

- 会话标题（可默认取主题或首次生成主题）
- 所属用户 ID
- 当前结果 ID（可选，也可通过最新结果反查）
- 创建时间
- 更新时间
- 状态（可选，第一版可不做）

### 4.6 教案记录管理

#### 功能范围

- 查看某个会话下的全部聊天记录
- 每条记录保存老师发送内容与 AI 回复内容
- 支持复制教案内容
- 支持编辑生成结果后保存
- 支持记录当次使用的大模型信息
- 支持在生成完成后直接进入教案编辑页

#### 记录字段建议

- 会话 ID
- 模板 ID
- llm_config_id
- model_name
- 用户消息内容
- AI 回复内容
- 输入内容 JSON
- 发送给模型的最终 prompt（建议保存，便于排查）
- 模型返回结果
- 编辑后结果内容
- 创建时间

## 5. 非功能需求

### 5.1 易用性

- 页面简单，操作路径短
- 移动端优先设计，单手可操作
- 页面需兼容 PC 端使用，桌面端仍需完整支持会话、生成、查看和编辑流程
- 生成按钮、模板切换、会话入口清晰

### 5.2 性能

- 普通列表接口响应建议在 1 秒内
- 大模型生成接口按外部 API 能力决定，前端需展示加载中状态

### 5.3 安全性

- 密码不可明文存储
- 用户数据隔离
- 大模型 API Key 保存在服务端，不暴露前端

### 5.4 可维护性

- 前后端分离
- 数据表结构简单清晰
- 项目启动后自动建表，降低初始化成本

## 6. 角色与权限

第一版仅考虑普通用户角色，不做后台管理端。

- 普通用户只能操作自己的模板、会话、教案记录

## 8. 接口草案

### 8.1 认证

- `POST /api/auth/register` 注册
- `POST /api/auth/login` 登录
- `GET /api/auth/me` 获取当前用户信息

### 8.2 模板

- `GET /api/templates` 模板列表
- `POST /api/templates` 新建模板
- `PUT /api/templates/{id}` 编辑模板
- `DELETE /api/templates/{id}` 删除模板
- `GET /api/templates/{id}` 模板详情

### 8.3 大模型

- `GET /api/llm-configs/options` 获取前端可选模型列表
- `GET /api/llm-configs/default` 获取默认模型

### 8.4 会话

- `GET /api/sessions` 会话列表
- `POST /api/sessions` 新建会话
- `GET /api/sessions/{id}` 会话详情
- `GET /api/sessions/{id}/messages` 查询会话聊天记录
- `DELETE /api/sessions/{id}` 删除会话

### 8.5 教案生成

- `POST /api/lesson-records/generate` 生成教案
- `GET /api/lesson-records/session/{sessionId}` 查询某会话下记录
- `PUT /api/lesson-records/{id}` 编辑并保存教案结果

### 9.2 数据库选型建议

建议第一版直接使用 PostgreSQL。

- 原因 1：原生支持 `json/jsonb`，适合保存生成输入参数
- 原因 2：后续如果要做检索、筛选、会话扩展，灵活性更高
- 原因 3：相比 MySQL，在这种“结构化字段 + 半结构化输入”的场景更顺手

如果你当前环境里 MySQL 更成熟，也可以继续用 MySQL，但第一版从开发体验看 PostgreSQL 更合适。

## 10. 技术栈建议

建议采用单体项目思路，不做严格前后端分离：后端使用 Spring Boot，前端使用 Vue 3，前端代码与后端放在同一仓库中，最终由 Spring Boot 直接托管打包后的 H5 静态资源。

### 10.1 整体架构建议

- 单仓库开发，前后端代码放在一个项目中管理
- 接口层仍然保留前后端边界，但部署时合并为一个服务
- Vue 打包后的静态文件由 Spring Boot 的 `static` 目录托管
- 第一版不拆微服务，不拆独立前端部署，降低复杂度

### 10.2 后端建议

- Java 17
- JDK 17
- Spring Boot 3
- Spring Web
- Spring Security + JWT
- MyBatis-Plus
- PostgreSQL
- Flyway 用于建表
- WebClient 调用大模型 API

### 10.3 前端建议

- Vue 3
- Vite
- TypeScript
- Pinia
- Vue Router
- UI 组件库建议选 Vant，适合 H5

### 10.4 为什么选 Vue，不选 React

- 你的既有经验是 Spring、Vue，直接复用学习成本最低
- 这个项目以表单录入、列表管理、详情查看为主，Vue 更贴合这类中后台和 H5 混合场景
- 第一版功能不复杂，Vue 3 + Vant 的组合可以更快落地
- 对当前需求来说，React 没有明显收益，反而会增加技术选择和项目初始化成本

### 10.5 部署建议

- 前端静态资源由 Spring Boot 直接托管
- 后端单体服务部署
- 数据库独立部署

## 11. 关于“启动自动建表”的建议

优先建议使用 Flyway。

- 原因 1：项目启动自动执行 SQL 迁移，满足“启动就自动创建表”
- 原因 2：比 Hibernate 自动建表更可控，后续字段调整不容易失控
- 原因 3：第一版表少，维护成本低

不建议只依赖 JPA 的 `ddl-auto=update` 作为正式方案，因为后续改表时容易产生不可控行为。第一版如果追求极简，也可以先用 Flyway 初始化一版基础表结构。

## 12. 大模型接入建议

### 12.1 Prompt 组装方式

建议后端统一组装：

- 系统约束：固定教案输出格式
- 用户模板：用户自定义模板内容
- 本次输入：主题、教具、活动目标等
- 会话上下文：同一会话最近若干轮生成记录
- 模型配置：用户当前所选模型的名称与接口参数

最终组合为一次完整 prompt 后调用模型接口。

### 12.2 需保存的数据

- 所选模板
- 当次使用的大模型配置
- 用户输入内容
- 最终 prompt
- 模型输出结果
- 模型名称（建议预留）

### 12.3 第一版范围控制

- 先只接 OpenAI Compatible（Responses）风格供应商
- 模型配置由后台统一维护，可直接通过数据库手动维护
- 前端只做模型展示与选择，不做模型配置管理页面
- 不做流式输出也可以，先做普通同步返回

### 12.4 当前默认模型接入信息

- provider：OpenAI Compatible（Responses）
- base_url：`http://192.168.12.22:7898/v1`
- 后端按兼容 OpenAI 风格接口设计，便于后续切换供应商
- 数据库中保存模型配置，不把模型信息写死在配置文件中

## 13. 第一版建议范围

建议第一版只做最小可用版本：

- 注册 / 登录
- 大模型列表展示、默认模型回填、模型选择
- 模板增删改查
- 会话新建、列表、详情、删除
- 选择模板并生成教案
- 保存生成记录
- 支持编辑生成结果后保存

第一版先不做：

- 富文本编辑
- 教案导出 Word / PDF
- 模板共享
- 多角色后台
- 消息流式输出

## 16. 已确认方案

- 登录方式：用户名 + 密码
- 会话模式：带上下文的连续生成
- 首页形态：首页直接展示会话列表
- 新建会话：先选择模板和模型，再通过底部聊天输入框发送消息开始对话
- 会话详情：按聊天记录展示老师提问和 AI 回答
- 教案输出：固定格式输出，严格分段
- 生成结果：每个会话对应一个当前结果，生成后自动保存，支持手动编辑后保存
- 教案入口：生成完成后可直接查看教案并进入编辑页
- 模型接入：兼容 OpenAI 风格接口，生成前需要选择大模型
- 模型配置方式：后台维护多个模型，数据库可手动配置，前端只展示和选择
- 默认模型：支持配置一个默认大模型并自动回填
- 默认接口地址：`http://192.168.12.22:7898/v1`
- Java 版本：JDK 17
- 技术栈建议：单仓库单体项目，Spring Boot 3 + Vue 3 + PostgreSQL + Flyway + MyBatis-Plus

## 17. 实施建议

如果准备直接开工，建议按下面顺序启动：

1. 先确定大模型供应商
2. 按单仓库单体项目初始化 Spring Boot 3 + Vue 3 + PostgreSQL + Flyway
3. 先完成数据表和认证模块
4. 再完成模板、会话、生成记录模块
5. 最后对接大模型 API 和 H5 页面
